섹션 2 — 향후 작업 제안

10건
01

배치 1급 객체화

우선순위: 상 규모: M

batchId 컬럼과 BatchDTO 타입이 스키마/공용 타입에 이미 정의돼 있으나 어떤 코드도 사용하지 않음. 이 서비스의 핵심 시나리오가 "1,000건 대량 변환"인데 배치 진행률·배치 단위 재시도·ZIP 일괄 다운로드가 없어 운영자가 작업을 개별 확인해야 함. 큐 공정성도 배치 vs 단건 명시 레인으로 발전 가능.

02

review 사유 구체화 — 듀얼 엔진 교차 검증

우선순위: 상 규모: M

현재 review 사유는 경고 코드명(rhwp_small_pdf_review 등)뿐이라 검수자가 "무엇을 봐야 하는지" 모름. review 작업만 H2Orestart로 한 번 더 변환해 페이지 수·텍스트 양을 비교하면 "rhwp 7p vs LibreOffice 14p — 레이아웃 손실 의심" 같은 사람이 읽는 사유 제공 가능. 서비스의 존재 이유("성공으로 표시됐지만 원본과 다름"을 잡는 것)에 직결.

03

실서버 품질 코퍼스 재검증

우선순위: 상 규모: S

보고서양식.hwp가 rhwp 0.7.0 PartialTable 한계로 실패했고, 실서버(사이드카 포함)에서 H2Orestart 폴백이 통과하는지 미확인. quality-corpus-report.sh 스크립트는 준비돼 있음 — 이번에 내장한 런타임 패키징의 실효 검증이기도 함.

04

워커 동시성 (WORKER_CONCURRENCY)

우선순위: 중 규모: S

워커가 한 번에 1건 처리 — 1,000건 배치는 rhwp 1초/건이어도 17분, 사이드카 폴백 3초/건이면 50분. claimNext가 이미 낙관적 잠금으로 멀티 워커 안전하므로 루프 N개 기동만으로 처리량 N배. rhwp/LO 프로세스 메모리만 고려하면 됨.

05

DLQ 가시화 — 실패 사유 컬럼 + UI 탭

우선순위: 중 규모: M

isPermanentFailure 분류가 도입됐지만 ConversionFailureReason이 DB 컬럼으로 저장되지 않고 에러 문자열에만 박혀 있음. 사유별 집계와 "재시도 불가" 탭이 있어야 대량 배치 실패의 패턴(암호 문서 비율 등)을 파악하고 고객 안내 가능.

06

/metrics 운영 지표

우선순위: 중 규모: M

서비스 기획서 KPI(변환 성공률, review 비율, P99 변환 시간, 가용성)를 실측할 수단이 없음. /api/stats는 누적 집계뿐 — 큐 깊이, 엔진별 성공률/지연 히스토그램의 시계열화가 필요. Prometheus 텍스트 포맷이면 의존성 없이 구현 가능.

07

E2E 회귀 CI 통합

우선순위: 중 규모: M

Playwright e2e 스택이 존재하나 RUN_E2E=1 수동 실행. 이번에 추가된 preview 사전 생성·/health/engines·review 필터는 브라우저 레벨 동작이라 단위 테스트만으로 회귀를 못 잡음. GitHub Actions에 compose 기동 후 e2e 실행 잡 추가.

08

인라인 모드 동시 변환 상한

우선순위: 하 규모: S

USE_QUEUE 미설정 시 업로드마다 즉시 변환 — 다중 사용자 동시 업로드에서 rhwp/LO 프로세스가 무제한 기동될 수 있음. 운영 권고는 USE_QUEUE=1이지만 잘못 설정한 배포를 위한 코드 레벨 방어선(세마포어 4개 등)이 없음.

09

rhwp 업스트림 추적 (지속 작업)

우선순위: 하 규모: S

rhwp 0.7.0 한계 3종(PartialTable 렌더 중단, NFD 자모 글리프 겹침, 자간 압축 시 공백 유실)이 문서화돼 있음. 버전 고정(==0.7.0)을 채택했으므로, 업스트림 릴리스 시 코퍼스 재실행 → 핀 갱신하는 절차 운영 필요.

10

Hancom SDK / Aspose 실계정 검증

우선순위: 하 규모: L

공공기관 고충실도 요구 대응용 상용 엔진 자리는 레지스트리에 마련돼 있으나(hancom/aspose 설정 시 체인 1순위) 실제 계정으로 검증된 적 없음. 도입 시 품질 등급 good의 신뢰도가 좌우됨.
