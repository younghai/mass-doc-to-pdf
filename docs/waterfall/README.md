> **버전:** v1.0  
> **작성일:** 2026-06-11  
> **프로젝트:** Mass Doc to PDF  
> **목적:** 폭포수(Waterfall) 소프트웨어 개발 생명주기(SDLC)에 따른 전체 문서 구조 및 흐름 정의

---

## 개요

본 문서는 **Mass Doc to PDF** 프로젝트의 소프트웨어 개발을 **폭포수(Waterfall) 모델** 관점에서 체계적으로 관리하기 위한 마스터 가이드입니다.

각 단계는 이전 단계의 산출물을 입력으로 받아 순차적으로 진행되며, 단계 간 역류(Backflow)는 공식 변경 제어 프로세스를 통해 관리합니다.

## 요구사항 추적 매트릭스 (RTM)

폭포수 모델의 핵심 원칙 중 하나는 **추적 가능성(Traceability)** 입니다.

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  요구사항    │ ──→ │   시스템    │ ──→ │   상세      │ ──→ │   소스      │
│  (SRS)      │     │   설계      │     │   설계      │     │   코드      │
│  FR-001     │     │   SAD       │     │   컴포넌트A  │     │   Module A  │
└──────┬──────┘     └──────┬──────┘     └──────┬──────┘     └──────┬──────┘
       │                   │                   │                   │
       └───────────────────┴─────────┬─────────┴───────────────────┘
                                     ▼
                            ┌─────────────┐
                            │  테스트케이스  │
                            │  TC-001     │
                            └─────────────┘
```

모든 요구사항은 고유 ID를 부여받으며, 설계 문서, 구현 코드, 테스트 케이스와 상호 참조됩니다.

## 문서 변경 관리 프로세스

폭포수 모델에서는 단계가 진행된 후의 변경은 공식적인 **변경 제어(Change Control)** 가 필요합니다.

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ 변경 요청  │ → │  영향도   │ → │  수정    │ → │  검증    │ → │  승인    │ → │  배포    │
│ (Issue)   │    │  분석    │    │          │    │          │    │          │    │          │
└──────────┘    └──────────┘    └──────────┘    └──────────┘    └──────────┘    └──────────┘
```

1. **변경 요청 (CR):** GitHub Issue 또는 공식 변경 요청서 제출
2. **영향도 분석:** 관련 문서, 코드, 테스트에 미치는 영향 평가
3. **수정:** 문서/코드 변경 수행
4. **검증:** 정확성 및 일관성 확인 (리뷰, 테스트)
5. **승인:** 담당자 또는 CCB(Change Control Board) 최종 승인
6. **배포:** 변경 사항 문서 반영 및 공지
