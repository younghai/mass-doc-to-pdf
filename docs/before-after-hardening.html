01

큐 아키텍처 — 죽어 있던 큐를 살림

BACKEND · INFRA

가장 큰 구조 변화. 워커 프로세스와 큐 코드는 있었지만 배포 설정 어디에도 USE_QUEUE=1이 없어서 실제로는 한 번도 동작한 적이 없는 상태였습니다.

BEFORE — 6/10 이전
  • API가 항상 인라인 fire-and-forget 변환 — 응답 직후 백그라운드 Promise
  • standalone 워커는 영원히 빈 큐만 폴링 (유령 프로세스)
  • API 재시작 시 진행 중이던 변환 전부 유실
  • docker compose에 worker 서비스 자체가 없음
AFTER — 6/11
  • compose 4개 + env.example 전부에 USE_QUEUE=1 명시
  • worker 서비스 신설 — API와 같은 이미지, worker-main.js 실행
  • visibility timeout(5분) + requeueStale로 워커 크래시 자동 복구
  • prod에서는 API healthy 이후에만 워커 기동
+ worker:
+   command: ["node", "dist/queue/worker-main.js"]
+   environment:
+     USE_QUEUE: "1"
+     WORKER_VISIBILITY_TIMEOUT_MS: 300000

02

재시도 예산 알고리즘 — 클레임은 리스다

ALGORITHM

워커가 작업을 잡을 때마다 attempts를 올리면, 크래시 후 재클레임도 "시도"로 계산됩니다. 워커가 3번 죽으면 변환을 한 번도 못 해보고 영구 실패하는 버그였습니다.

BEFORE
// claimNext()
data: {
  status: "running",
  attempts: { increment: 1 },  // 크래시 리클레임도 소진
  lockedAt: now, lockedBy: workerId,
}
  • 크래시 → 리클레임 = 재시도 예산 1 소모
  • 최대 3회 크래시면 변환 시도 0번으로 영구 실패
AFTER
// claimNext() — 리스만 잡는다, attempts 불변
data: { status: "running", lockedAt: now, lockedBy: workerId }

// retryOrGiveUp() — 실패가 확정된 곳에서만
attempts: { increment: 1 }  // 실제 변환 실패만 계산
  • 크래시·리클레임은 무료 — 실패만 예산 소모
  • 회귀 테스트 신규: "crash reclaims do not burn retry budget"

SQLite 다중 프로세스 동시성

BACKEND

큐가 살아나면 API와 워커 두 프로세스가 같은 DB 파일에 동시 기록합니다. 기본 저널 모드에서는 즉시 SQLITE_BUSY가 터집니다.

BEFORE
  • 기본 rollback journal — 쓰기 잠금 충돌 시 즉시 에러
  • busy 대기 없음 → 간헐적 500
AFTER
PRAGMA journal_mode=WAL;
PRAGMA busy_timeout=5000;
  • 읽기·쓰기 동시 진행 (WAL)
  • 잠금 충돌 시 5초까지 대기 후 진행

11

품질 게이트 알고리즘 — pdfPageCount

ALGORITHM

PDF 1.5+는 XRef 스트림으로 오브젝트를 압축할 수 있어 /Type /Page 문자열이 본문에 보이지 않습니다. 기존 로직은 이런 정상 PDF를 "0페이지 = 저품질"로 오판했습니다.

BEFORE
count = matches(/\/Type\s*\/Page[^s]/g)
// PDF 1.5+ 압축 스트림 → 0 매치 → 멀쩡한 PDF가 저품질 판정
AFTER
// 1순위: Pages 트리 /Count N의 최댓값 (루트 노드 총계)
max(/\/Count\s+(\d+)/g)
// 폴백: 구형 PDF는 기존 /Type /Page 정규식
