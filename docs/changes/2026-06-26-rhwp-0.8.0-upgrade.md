# rhwp 엔진 업데이트 (rhwp-python 0.8.0 + rhwp-cli v0.7.17)

2026-06-26. 업스트림 `edwardkim/rhwp`의 누적 패치(v0.7.14~v0.7.17)와 PyPI
`rhwp-python` 0.8.0을 반영했다. 렌더 품질(수식·표·차트) 향상과 CLI 설치
재현성·단순화가 목적이다.

## 배경 (업스트림 격차)

- 기존 핀: `rhwp-python==0.7.0` (2026-06-04), `rhwp-cli`는 `main` 브랜치 무핀.
- 최신: `rhwp-python==0.8.0` (2026-06-21), 릴리스 `v0.7.17` (2026-06-22).
- 0.7.0 이후 흡수된 렌더 개선(서버측 PDF 변환에 직접 영향):
  - 수식 렌더링/배치 정밀화, 미주 흐름·간격 (v0.7.14/0.7.15)
  - 표 셀 내부 그림·회전 셀 배치 한컴 정합 (v0.7.14/0.7.16)
  - 차트 2D 근사 렌더 — "차트 (미지원)" placeholder 제거, OOXML 27종 중 13종
    렌더 (v0.7.17)
  - native PDF export API, Text IR v2 폰트 fallback 권위 (v0.7.16)

## 변경

### rhwp-python 0.7.0 → 0.8.0
- `apps/api/Dockerfile` — pip 핀 0.8.0.
- `standalone/scripts/install-ubuntu.sh` — `RHWP_PYTHON_VERSION` 기본값 0.8.0.

호환성은 wheel을 직접 풀어 검증했다:
- **API 동일**: `export_pdf(output_path)` / `render_pdf()` 시그니처 변화 없음
  (`rhwp_worker.py` 무수정). 0.8.0이 추가한 건 `verify_hwpx_roundtrip` 하나뿐
  (PDF 변환과 무관).
- **freetype 워크어라운드 유지**: 0.8.0의 manylinux wheel은 0.7.0과
  **바이트 동일한** `rhwp_python.libs/libfreetype-c0e61f0c.so.6`를 번들한다.
  `FT_Palette_Select` 심볼 재지정 패치는 여전히 필요하며 glob도 그대로 매칭된다.
- **신규 의존성 없음**: `pydantic<3,>=2.5`는 0.7.0에도 코어 의존성이었다(pip가
  자동 설치).

### rhwp-cli: `main` 무핀 → `v0.7.17` 태그 + 프리빌트 바이너리
- `standalone/scripts/install-rhwp-cli.sh` 재작성:
  - `RHWP_CLI_VERSION` 기본값 `v0.7.17`.
  - 기본 경로를 **공식 프리빌트 릴리스 바이너리 다운로드 + SHA256SUMS 검증**으로
    전환 → Rust 툴체인 불필요.
  - 폴백: 프리빌트가 없는 플랫폼(예: linux-aarch64), 체크섬 불일치,
    `RHWP_CLI_FROM_SOURCE=1`, 또는 `VERSION=main`이면 기존 cargo 소스 빌드.
  - `RHWP_CLI_URL` 오버라이드 경로는 그대로 유지.
- `standalone/env.example` — 설치 방식 주석 갱신.

검증(macOS arm64, 실제 실행):
- `rhwp-v0.7.17-macos-aarch64.tar.gz` 다운로드 → sha256 검증 → 설치 성공.
- `rhwp --version` → `rhwp v0.7.17` (preflight `--version` 프로브 호환 확인).
- `export-pdf <파일.hwp> [-o 출력.pdf]` 서브커맨드 존재 확인
  (`RhwpCliConverter` 호출과 일치).

### `/health/engines`에 렌더러 버전 노출 (관측성)
- `apps/api/src/convert/preflight.ts`:
  - `EngineProbe`에 `version?` 추가. `/health/engines`의 `preflight` 응답에 자동
    포함된다(app.ts 무수정).
  - `ProbeRunner`가 stdout(`Promise<string>`)을 반환하도록 변경.
  - rhwp 프로브를 `import rhwp; print(getattr(rhwp,'rhwp_core_version',lambda:'')())`로
    바꿔 코어 버전을 캡처(getattr 가드 — 구버전 wheel도 exit 0 유지). rhwp-cli는
    `--version` 출력에서 semver를 파싱.
  - `logEnginePreflight`가 부팅 시 감지된 버전을 로그로 남긴다.
- `apps/api/src/convert/preflight.test.ts` — 러너 시그니처 갱신 + 버전 캡처 테스트
  추가. 전체 API 스위트 135 통과.

### Python 워커 폰트 처리 정정
- `apps/api/src/convert/workers/rhwp_worker.py` — `_apply_font_paths()`가 시도하는
  `set_font_paths`/`set_font_dirs`/`add_font_paths`는 0.7.0·0.8.0 wheel 어디에도
  없어 항상 no-op이고, **rhwp 코어는 `RHWP_FONT_PATHS` env를 읽지 않는다**(네이티브
  `.so` 문자열 검사로 확인 — `ttfs/hwp`는 있으나 `RHWP_FONT_PATHS`는 없음). 실제
  폰트는 `engines/rhwp.ts`가 cwd에 만드는 `ttfs/hwp/` 심링크 + 시스템 fontconfig로
  해결된다. 오해를 부르던 "env only" 메시지와 docstring을 실제 메커니즘에 맞게
  정정했다(전방호환용 hasattr 프로브는 유지).

## 런타임 검증 (2026-06-26, macOS arm64 / Python 3.11)

`rhwp_worker.py`로 실제 코퍼스 문서 11건(`tmp/preview-objects/.../src`)을 0.7.0과
0.8.0 venv에서 각각 변환해 비교했다.

- **0.8.0 정상 동작**: `import rhwp` OK, 패키지 0.8.0 / **코어 0.7.16**
  (0.7.0은 코어 0.7.13). 즉 범프로 렌더러 코어가 0.7.13 → 0.7.16으로 올라간다.
- **회귀 없음**: 0.7.0에서 변환되던 9건은 0.8.0에서도 전부 변환 성공.
- **HWPX 출력 변화**: 사업계획서.hwpx 산출 PDF 98434B → 98441B — 코어 렌더 변경
  반영(나머지 PDF는 바이트 크기 동일).
- **OPEN-001 미해결 확인**: "보고서 양식.hwp"(14KB)는 0.7.0·0.8.0 **양쪽 모두**
  `LAYOUT_OVERFLOW: type=PartialTable, bottom=0.0`로 실패. 0.8.0 업그레이드로
  해결되지 않음 → OPEN-001 갱신(test-results.md).

### Linux/Docker freetype 워크어라운드 검증 (2026-06-27, container)

`node:20-slim`(linux/aarch64) 컨테이너에서 Dockerfile의 rhwp 런타임 설치 절차를
그대로 재현해 실검증했다. macOS는 CoreText라 freetype 경로가 실행되지 않으므로
이 단계가 타깃(arm64 manylinux wheel) 검증의 핵심이다.

- **워크어라운드 전**: `import rhwp` →
  `ImportError: ... undefined symbol: FT_Palette_Select` (버그 재현 — 0.8.0에서도
  발생).
- **워크어라운드 후**(번들 freetype을 distro freetype으로 symlink): `import OK |
  core 0.7.16`.
- **실제 변환**: 코퍼스 HWP를 `export_pdf`로 변환 성공(656,943 bytes).

→ Dockerfile/installer의 freetype symlink 워크어라운드는 0.8.0 + Linux arm64에서
정상 동작하며, 정밀 엔진이 H2Orestart로 폴백되지 않고 실제로 로드된다.

## 후속 (범위 밖)

1. **품질 코퍼스 시각 충실도 회귀** — 기능·오버플로·Linux import는 위에서 실증
   완료. 남은 건 폰트 임베딩 등 **시각 품질 비교**(Docker 이미지 빌드 후
   `docs/changes/2026-06-11-hwp-quality-corpus.md` 코퍼스 기준).
2. **OPEN-001 상류 보고** — PartialTable 오버플로는 코어 0.7.16에도 잔존
   (test-results.md OPEN-001 갱신 완료). 상류 추적 결과: rhwp는 PartialTable/
   LAYOUT_OVERFLOW를 문서별로 활발히 수정(다수 closed, 최근 #1488 2026-06-25)하나,
   우리 시그니처(`bottom=0.0`, 높이 0 표 영역)와 일치하는 **열린/닫힌 이슈는 없음**
   → 미보고로 판단. 해결 경로는 "보고서 양식.hwp" 최소 재현을 상류에 리포트 후
   다음 릴리스에서 재테스트, 또는 Hancom sidecar 대안.
