## P1 — 서버 작업 큐 + 워커 (진행 중)

> 병행 quality 작업과 충돌 회피를 위해 **신규 파일 위주**로 구현. hot 파일(convert.ts/registry.ts/shared/types.ts/quality.ts/jobs.ts)은 **무수정**.

### 변경 항목 (신규)
- `apps/api/prisma/schema.prisma` + 마이그레이션 `20260607022702_queue_durability` — `ConversionJob`에 큐 내구성 컬럼(`qualityMode`, `batchId`, `attempts`, `lockedAt`, `lockedBy`) + 인덱스(`status, lockedAt`). 모두 추가/nullable이라 기존 행 안전
- `apps/api/src/queue/jobQueue.ts` — **DB 기반 durable 큐**(Redis 불필요 → standalone 호환): `enqueue`, 원자적 `claimNext`(낙관적 가드), `release`, `retryOrGiveUp`(maxAttempts 재시도), `requeueStale`(크래시 워커 복구), `depth`
- `apps/api/src/queue/processConversion.ts` — claim된 잡을 스토리지 소스로부터 변환 → PDF/품질리포트 저장 → markSuccess (quality 모듈 read-only 재사용)
- `apps/api/src/queue/worker.ts` — `runWorkerOnce`(성공=release / 실패=retryOrGiveUp→markFailed) + `runWorkerLoop`(idle 백오프 + 주기적 stale 복구)
- `apps/api/src/queue/worker-main.ts` — 워커 **독립 프로세스** 엔트리포인트(SIGINT/TERM graceful stop)
- `standalone/systemd/mass-doc-to-pdf-worker.service.in` + `install-systemd.sh`(워커 유닛 생성·enable) + `env.example`(WORKER_*) — 워커를 별도 systemd 서비스로

### 검증
- `jobQueue.test.ts`(7) + `worker.test.ts`(3) 통과 — FIFO claim, 중복 claim 방지, 재시도/포기, stale 복구, 성공 시 PDF+리포트 저장
- API 전체 17파일 70통과(+1 skip), 타입체크 clean

### 남은 단계 (조율 필요 — hot 파일)
- `convert.ts`의 `void finishConversion(...)`(인-프로세스) → **`queue.enqueue(job.id)`** 로 전환 + `app.ts`/`server.ts` deps에 큐 주입. convert.ts는 병행 편집 중이라 충돌 방지를 위해 **마지막에 합의 후 적용**.
